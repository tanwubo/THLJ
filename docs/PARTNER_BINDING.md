# 情侣绑定数据说明

本文说明当前版本中，账号 A 和账号 B 进行情侣绑定时，系统如何决定共享哪一份数据，以及绑定、解绑前后数据会发生什么变化。

## 一句话结论

当前版本下，**谁的邀请码被输入，谁当前的数据就会成为绑定后的共享主数据**。

换句话说：

- A 输入 B 的邀请码完成绑定：绑定后双方共用 **B 当前的数据**
- B 输入 A 的邀请码完成绑定：绑定后双方共用 **A 当前的数据**

系统目前不会在绑定时自动合并 A 和 B 两边的历史数据。

## 当前规则

### 1. 绑定成功后，双方看到的是同一份数据

绑定前：

- A 有自己的一份数据
- B 有自己的一份数据

绑定后：

- A 和 B 都会读取同一份共享数据
- 任意一方新增、修改、删除内容，另一方会看到相同结果
- 时间线、待办、费用、备忘录、附件都会进入同一数据空间

### 2. 共享主数据由“邀请码归属方”决定

系统判断规则不是让用户在界面上单独点选“以谁的数据为准”，而是通过绑定动作本身确定：

- 输入谁的邀请码，就以谁当前的数据作为绑定后的共享主数据

这意味着，当前版本里“如何选择主数据”的实际操作方式就是：

1. 先确认你们想保留谁目前的数据
2. 由另一方输入这位用户的邀请码完成绑定

### 3. 另一方原有数据不会被并入共享数据

例如：

- A 当前有一套自己的筹备数据
- B 当前也有一套自己的筹备数据
- A 输入 B 的邀请码并绑定成功

那么绑定后：

- 双方看到的是 B 的数据
- A 原先的数据不会并入 B 的数据
- 系统也不会自动把 A、B 两边的节点、待办、费用做合并去重

因此，绑定动作本质上是：

- 从“两套独立数据”
- 切换为“共用其中一套数据”

## 用户如何选择保留哪一份数据

当前版本没有单独的“选择主数据”按钮，用户需要按下面的方式操作。

### 场景 1：希望保留 A 的数据

操作方式：

1. 打开 A 的设置页，复制 A 的邀请码
2. 让 B 登录自己的账号
3. 在 B 的设置页输入 A 的邀请码
4. 完成绑定

结果：

- 绑定后共享主数据是 A 当前的数据
- A 和 B 后续都看到 A 这份数据

### 场景 2：希望保留 B 的数据

操作方式：

1. 打开 B 的设置页，复制 B 的邀请码
2. 让 A 登录自己的账号
3. 在 A 的设置页输入 B 的邀请码
4. 完成绑定

结果：

- 绑定后共享主数据是 B 当前的数据
- A 和 B 后续都看到 B 这份数据

## 建议的用户操作指引

在实际使用中，建议按以下顺序操作：

1. 先确认双方目前哪一边的数据更完整、更新、更值得保留
2. 以“保留这份数据”为目标，决定应该输入谁的邀请码
3. 绑定前各自先做一次数据备份
4. 再执行绑定

建议直接按下面这句话给用户说明：

> 如果你想保留谁当前的筹备数据，就输入谁的邀请码进行绑定。

## 绑定时数据变化明细

绑定成功后，系统会发生这些变化：

### 用户关系变化

- A 的 `partner_id` 指向 B
- B 的 `partner_id` 指向 A

### 数据归属变化

- 双方会指向同一个共享数据归属
- 后端后续读取数据时，不再按“当前登录账号自己的数据”读取
- 而是按“这对情侣当前的共享主数据”读取

### 业务数据表现

以下内容都会改为读取同一份：

- 时间线节点
- 待办事项
- 费用记录
- 备忘录
- 附件

### 实时同步表现

绑定成功后：

- A 修改数据，B 会看到最新结果
- B 修改数据，A 也会看到最新结果

这里的前提不是“双方各改各的，再互相同步合并”，而是“双方本来就在操作同一份数据”。

## 解绑时数据会怎样

解绑不是简单取消情侣关系。

当前版本中，解绑流程是：

1. 先以解绑当下的共享数据为基础
2. 把这份共享数据恢复为双方各自独立的一份
3. 再解除绑定关系

解绑完成后：

- A 有自己的一份数据
- B 有自己的一份数据
- 两边初始内容都来自解绑当下的最新共享数据
- 解绑后双方不再实时同步

可以理解为：

- 绑定时：两份数据收敛成一份共享数据
- 解绑时：一份共享数据再拆回两份独立数据

## 需要特别提醒用户的风险

### 1. 绑定不是“自动合并双方历史数据”

当前版本最容易被误解的点就是这里。

系统不会把：

- A 的时间线节点
- B 的时间线节点
- A/B 的待办、费用、附件

自动合并成一套更完整的新数据。

当前规则是“保留一方当前数据作为共享主数据”。

### 2. 绑定前应先确认保留哪一边

如果用户没先商量清楚谁的数据应该保留，就很容易出现：

- 实际绑定成功了
- 但发现共享后的数据不是自己以为会保留的那一份

### 3. 绑定前建议先备份

虽然系统支持导出备份，但用户如果在绑定前没有备份，后续想回头精确恢复“绑定前各自原始状态”，成本会更高。

## 对外文案建议

如果需要在产品页面、帮助中心或弹窗中解释，可以直接使用下面这段：

> 绑定后，双方将共用同一份筹备数据。  
> 当前版本下，输入谁的邀请码，就会以谁当前的数据作为绑定后的共享数据。  
> 系统不会自动合并双方历史数据。  
> 建议先确认要保留哪一方的数据，并在绑定前先完成备份。

## 示例

### 示例 1

- A 的数据很完整
- B 刚注册，几乎没有内容

正确做法：

- 让 B 输入 A 的邀请码

结果：

- 绑定后双方共用 A 的数据

### 示例 2

- B 一直在维护时间线和预算
- A 只是补充了少量内容

正确做法：

- 让 A 输入 B 的邀请码

结果：

- 绑定后双方共用 B 的数据

## 当前版本的限制

当前版本还不支持以下能力：

- 绑定时让用户在界面上显式选择“保留 A”或“保留 B”
- 绑定时自动合并双方历史数据
- 绑定前展示 A/B 两边数据差异对比，再确认合并策略

如果后续产品要降低用户理解成本，建议增加“绑定确认页”，明确展示：

- 当前将保留哪一方数据
- 另一方数据不会自动合并
- 用户是否要继续
