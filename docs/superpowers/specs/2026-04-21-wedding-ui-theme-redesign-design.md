# 婚嫁管理系统 UI 主题化重设计方案

## 背景

当前前端已经具备完整主流程能力，包括登录、时间线、统计、节点详情与设置，但界面仍偏原型化与工具化，存在以下问题：

- 视觉语言不统一，不同页面的头部、卡片、按钮和导航缺少一致性。
- 页面大量直接使用写死颜色与 emoji，品牌感弱，也不利于后续做主题切换。
- 交互层级较粗糙，主次操作、重点信息和辅助信息的视觉权重不稳定。
- 主流程页面缺少“婚礼筹备”场景应有的仪式感和高级感，当前体验更接近通用移动后台。

本次改造目标不是增加新业务能力，而是在不破坏现有业务流程的前提下，把主流程重塑为“高定婚礼计划册”风格的移动端工作台，并建立可持续扩展的主题系统。

## 目标

### 产品目标

- 将主流程整体升级为具有统一品牌气质的婚礼筹备应用。
- 默认主题采用“中式雅红金”方向，强调仪式感、珍贵感与高级感，但避免俗气的大红大金。
- 主题切换作为正式能力进入产品，首期实现“统一布局 + 可切换视觉主题”的基础架构。
- 设置页提供主题切换入口，切换后实时生效并本地持久化。

### 工程目标

- 建立可维护的 Design Token 体系，移除主流程页面中的关键硬编码色值依赖。
- 抽出可复用的页面外壳与基础视觉组件，避免每个页面各自定义一套样式。
- 保持现有业务逻辑、接口调用、状态管理和测试主体可用，不对业务数据流做结构性重写。

## 非目标

- 不在本次引入新的业务模块或新的路由页面。
- 不做“主题切换后页面结构也变化”的多套布局系统。
- 不进行大规模状态管理重构，不把频繁变化的业务数据塞入主题上下文。
- 不追求桌面端后台化布局，仍以移动端优先为主，桌面端做增强适配。

## 总体设计方向

### 品牌定位

产品定位为“高定婚礼计划册”，而不是传统任务管理工具。视觉上参考高端婚礼请柬、宴会手册与精品品牌画册的感觉，强调：

- 庄重但不压迫
- 精致但不繁复
- 喜庆但不俗艳
- 情绪价值强，但不牺牲日常使用效率

### 主题策略

首期采用“统一结构，多主题换肤”策略。

- 页面布局、组件结构、信息顺序在不同主题下保持一致。
- 主题差异仅体现在颜色、背景、边框、阴影、文字气质和局部装饰表达上。
- 默认主题为 `雅红金`。
- 第二主题预留为 `奶油香槟`，本次重点完成主题架构与默认主题落地；第二主题可在同一套 token 体系下补齐。

推荐方案采用 `Design Token -> 全局主题类 -> 页面消费 Token` 的链路，而不是在页面内部写条件分支样式。

## 视觉系统

### 默认主题：雅红金

默认主题采用克制的中式婚礼视觉：

- 主品牌色：深酒红，承担头部品牌区、主按钮、激活态、关键进度强调。
- 金属强调色：雾金或旧金，只用于描边、数字强调、分隔线和轻装饰，不做大面积底色。
- 内容背景：暖米白、淡绢白或轻微暖灰白，保证长时间阅读舒适。
- 辅助层：浅金雾、柔和粉棕或低饱和胭脂色，用于丰富层次，但不直接替代主品牌色。

视觉原则：

- 深色集中用于“封面区”与少数重点区域。
- 阅读区和表单区保持浅底高对比。
- 金色必须克制，避免出现婚庆模板式的廉价质感。

### 第二主题预留：奶油香槟

第二主题保留现代婚礼方向：

- 主色更偏香槟金、奶油白、柔和玫瑰棕。
- 结构与组件不变，仅切换 token。
- 该主题作为架构预留项，不要求本次在视觉细节上做到与默认主题同级别精修。

### 字体策略

正文与功能信息优先中文可读性：

- 正文字体采用稳定的中文无衬线方案，如 `Noto Sans SC` 或等价安全回退方案。
- 标题、重点数字、封面式区块可使用更具仪式感的 serif 风格字体。
- 不使用大面积花体，不把标题字体应用到表单、正文、数据表述和高频交互区。

### 图标策略

- 移除底部导航和状态中的 emoji。
- 统一改为 SVG 图标体系，优先使用同一套风格一致的图标。
- 图标只承担语义提示，不能替代文案。

## 主题架构设计

### Token 分层

主题至少应覆盖以下 token 分类：

- 背景：`bg-base`、`bg-muted`、`bg-elevated`、`bg-hero`
- 文字：`text-primary`、`text-secondary`、`text-muted`、`text-on-brand`
- 品牌：`brand-primary`、`brand-primary-soft`、`brand-accent`
- 状态：`status-success`、`status-warning`、`status-neutral`
- 边框：`border-soft`、`border-strong`
- 阴影：`shadow-card`、`shadow-floating`
- 装饰：`ornament-line`、`ornament-glow`

### 主题挂载方式

- 在根节点挂载主题标识，例如 `data-theme="ceremony-red"`。
- 主题变量通过全局 CSS 或等价机制注入，不在页面中硬编码条件判断。
- 页面内样式优先消费语义 token，而不是直接使用颜色值。

### 持久化策略

- 设置页切换主题后，将主题 key 持久化到本地存储。
- 应用初始化时读取并应用上次主题。
- 若本地无设置，默认回退到 `ceremony-red`。

## 页面改造方案

### 登录页

目标：将当前功能表单升级为品牌入口页。

改造点：

- 增加品牌封面式头图区域，承载产品名、副标题和婚礼计划氛围。
- 登录表单置于独立卡片表面，形成“请柬内页”效果。
- 按钮、输入框和辅助文案遵循全局主题 token。
- 在不降低登录效率的前提下，强化进入产品时的品牌印象。

### 时间线页

目标：成为主品牌视觉与日常使用频率最高的主工作台。

改造点：

- 页面顶部升级为品牌概览区，展示产品标题、当前用户身份与总览信息。
- 节点列表采用统一的高质感卡片样式，突出节点标题、状态、截止日期与进度。
- 创建节点入口改造成更明确的主操作区域，而不是普通虚线占位按钮。
- 编辑、删除与状态切换等次级操作进行层级收敛，避免视觉噪音。
- 底部导航统一为品牌化 tab bar。

### 统计页

目标：从普通数据页升级为“筹备概览”页面。

改造点：

- 顶部延续品牌头图区，但比时间线更简洁。
- 圆环进度、节点状态卡片、待办统计统一到同一视觉系统。
- 强化关键数字和完成率的精致呈现，但保留快速扫读能力。
- 避免做成传统 BI 或后台图表风格。

### 节点详情页

目标：成为某个筹备节点的聚焦工作台。

改造点：

- 顶部区域强化节点标题、状态与摘要信息，形成“章节首页”感。
- 待办列表、备忘录、资金摘要和节点操作拆分为清晰的信息区块。
- 卡片层级和区块标题风格统一，突出主任务，弱化杂项操作。
- 保持现有待办新增、编辑、状态切换、删除、备忘录保存等行为不变。

### 设置页

目标：承担主题切换与个人设置能力。

改造点：

- 增加“外观主题”区，提供主题预览与切换入口。
- 保留现有账号、配对或相关设置能力。
- 页面视觉遵循主流程统一语言，避免像附属管理页。

## 组件层设计

### App Shell

为主流程页面提供统一页面外壳，负责：

- 安全区与最小高度
- 页面背景层
- 顶部品牌区域容器
- 内容区最大宽度与垂直节奏
- 底部导航占位和安全区留白

### Brand Header

统一顶部品牌区的设计语言，但允许不同页面控制信息密度：

- 时间线：品牌感最强
- 节点详情：突出当前节点与摘要
- 统计：强调总览和关键数据
- 设置：相对克制

### Surface Card

作为全局内容容器，用于：

- 节点卡片
- 数据摘要卡
- 表单卡
- 设置项分组
- 空状态与提示块

共同特征：

- 浅底高对比
- 柔和阴影
- 细边框或发光描边
- 较大圆角
- 清晰内边距节奏

### Status Pill

统一状态标签表达：

- 不再使用 emoji
- 使用颜色、描边和文案表达状态
- 需保证轻量且可扫读
- 不同主题下仍保留一致语义映射

### Button System

至少区分三类按钮：

- 主按钮：创建、保存、完成
- 次按钮：普通编辑与辅助操作
- 幽灵按钮：低优先级动作和工具性操作

按钮体系必须保证：

- 触达面积足够
- 激活态和禁用态明确
- 与主题 token 解耦

### Bottom Navigation

底部导航使用标准可访问结构与 SVG 图标：

- 时间线
- 统计
- 设置

激活态通过品牌色、轻背景和文字权重共同表达。

## 交互与动效规范

- 动效仅用于提升层次感和响应反馈，时长控制在 150ms 到 300ms。
- 禁止持续性的装饰动画，避免分散注意力。
- 卡片悬浮、页面进入、进度变化可使用轻量过渡。
- 所有交互元素使用语义化标签，如 `button`、`nav`、`main` 等。
- 键盘焦点可见，不以颜色作为唯一状态提示。
- 移动端输入遵守对应输入方式，例如数字和日期输入使用合适的输入模式。

## 响应式与适配原则

- 采用移动端优先设计。
- 重点验证 375px、768px、1024px、1440px 四档体验。
- 手机端保持单栏高密度信息流。
- 平板及桌面端主要做留白增强、容器宽度控制和卡片栅格优化，不改业务布局逻辑。
- 底部导航与固定区域需考虑安全区和内容遮挡问题。

## 可访问性要求

- 浅底内容区必须保持足够文字对比度。
- 焦点态清晰可见。
- 图标、颜色和文案联合表达状态，不能只靠颜色区分。
- 表单保留标签或明确占位说明。
- 尽量避免因主题切换导致可读性下降。

## 技术实施原则

- 保持 React 页面逻辑主体不变，优先在样式和布局层完成重构。
- 主题相关状态使用 Context 或等价轻量全局能力承载，不放入频繁变化的业务 store。
- 主流程的表单仍保持受控组件模式。
- 页面重构时优先抽取基础 UI 层，而不是在每个页面重复复制样式结构。

## 风险与约束

### 风险

- 节点详情页功能密度高，视觉重构时最容易引入交互回归。
- 主题 token 替换过程中，如果存在大量散落的写死色值，容易出现漏改。
- 使用装饰性字体和深色品牌区时，若控制不当会影响中文界面的日常可读性。

### 应对策略

- 优先建立基础主题层与页面外壳，再逐页替换视觉。
- 对核心交互路径进行回归验证，尤其是待办、备忘录、节点状态更新。
- 保持阅读区浅底高对比，把视觉表达集中在头图区、按钮和强调信息。

## 验收标准

- 登录、时间线、统计、节点详情、设置五个主流程页面完成统一视觉改造。
- 产品默认呈现“高定婚礼计划册”风格，且默认主题为雅红金。
- 应用具备正式的主题切换能力，并在设置页可操作。
- 页面主要样式依赖 token，而不是散落的硬编码颜色。
- 底部导航与状态表达移除 emoji，替换为统一图标和视觉系统。
- 现有主流程业务行为保持可用。
- 移动端体验优先，桌面端无明显布局崩坏与横向滚动。

## 后续扩展建议

- 在主题体系稳定后补齐奶油香槟主题的精修视觉。
- 若后续需要节日主题、订婚主题或仪式阶段主题，可继续沿用同一 token 架构扩展。
- 若产品继续增长，可在此基础上逐步沉淀为独立的应用设计系统。
