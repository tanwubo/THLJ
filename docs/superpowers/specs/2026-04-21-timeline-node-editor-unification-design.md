# 时间线节点编辑器统一设计

## 背景

当前 `frontend/src/pages/Timeline/Timeline.tsx` 中，“创建节点”和“编辑节点”虽然操作的是同一组核心字段，但容器结构和交互表现不一致：

- `创建节点` 使用 antd-mobile `Dialog` 默认动作区，视觉上更接近轻量确认弹窗。
- `编辑节点` 已经区分移动端和桌面端：
  - 移动端使用底部抽屉式 `Popup`
  - 桌面端使用 `Dialog`
- 两者表单结构重复，后续改字段、改样式、改文案时容易漂移。

用户目标不是只修复创建节点样式，而是将“创建节点”和“编辑节点”统一成同一套样式与结构，同时保留编辑节点现有的端差异处理方式。

## 目标

- 让 `创建节点` 和 `编辑节点` 共享同一套视觉语言、内容结构与按钮布局。
- 保留现有端差异策略：
  - 移动端使用底部抽屉式容器
  - 桌面端使用居中对话框容器
- 将重复的表单字段与弹窗壳子提炼成复用组件，降低 `Timeline.tsx` 中的重复 JSX。
- 不改变节点创建、编辑、保存、关闭的业务行为。

## 非目标

- 不改动节点卡片的行内编辑、状态快捷切换、截止日期快捷编辑逻辑。
- 不调整节点字段范围，仍然只处理名称、描述、截止日期、状态。
- 不对 `NodeDetail` 页或其他弹窗做连带重构。
- 不引入新的状态管理方式。

## 方案对比

### 方案 1：抽出统一容器组件与统一表单组件

- 新增 `NodeEditorModal`
- 新增 `NodeEditorForm`
- `Timeline.tsx` 只保留创建/编辑的状态和提交逻辑

优点：

- 容器、头部、按钮区和端差异逻辑集中管理
- 表单字段只维护一份
- 创建与编辑在视觉和结构上天然同步
- 后续继续调整节点编辑体验时，改动面最小

缺点：

- 相比只抽表单，改动稍大，需要重新整理 `Timeline.tsx` 中当前的弹窗 JSX

### 方案 2：只抽表单内容

- 创建与编辑各自保留容器
- 仅复用字段区

优点：

- 修改量较小

缺点：

- 容器和按钮区仍然重复
- 当前最明显的不一致仍然会被保留在结构层

### 方案 3：保留现状，仅补样式

优点：

- 改动最小

缺点：

- 只能暂时“看起来更像”
- 结构和交互复用问题仍然存在，维护成本最高

## 选定方案

采用方案 1：抽出统一容器组件 `NodeEditorModal` 与统一表单组件 `NodeEditorForm`。

这是唯一能同时满足“创建/编辑统一样式”和“保留端差异处理”两个目标的方案，而且不会把重复逻辑继续留在 `Timeline.tsx` 内。

## 交互设计

### 端差异策略

- 移动端：使用 `Popup` 作为底部抽屉式容器
- 桌面端：使用 `Dialog` 作为居中弹窗容器

该策略沿用现有编辑节点的设备差异处理逻辑，创建节点接入同一模式，而不是另起一套实现。

### 创建节点

- 标题：`创建节点`
- 眉标：`New Node`
- 说明文案：沿用现有创建语义，强调先定义名称、说明与时间
- 字段：
  - 节点名称
  - 描述
  - 截止日期
- 不显示状态字段，保持现有创建流程和默认状态逻辑不变
- 底部操作：
  - 左侧 `取消`
  - 右侧 `创建`

### 编辑节点

- 标题：`编辑节点`
- 眉标：`Complete Edit`
- 说明文案：沿用现有编辑语义，强调在一个面板内统一调整字段
- 字段：
  - 节点名称
  - 描述
  - 截止日期
  - 节点状态
- 底部操作：
  - 左侧 `取消`
  - 右侧 `保存`

### 关闭与提交行为

- 点击遮罩可关闭
- 点击取消关闭，不提交
- 点击主操作按钮执行现有的创建或保存逻辑
- 失败提示、成功提示、刷新时间线、实时事件发送逻辑维持现状

## 组件设计

### `NodeEditorForm`

职责：

- 只负责渲染字段区
- 根据 `mode` 或显式参数决定是否展示状态字段
- 不直接处理提交和关闭

输入建议：

- `mode: 'create' | 'edit'`
- `form`
- `onNameChange`
- `onDescriptionChange`
- `onDeadlineChange`
- `onStatusChange`

输出：

- 纯受控表单，无内部业务状态

结构要求：

- 保持现有 `timeline-node-dialog`、`timeline-node-dialog__intro`、`timeline-node-dialog__fields` 等视觉类名语义
- 文案和字段顺序在创建、编辑两种模式下保持一致

### `NodeEditorModal`

职责：

- 统一渲染外层容器
- 根据 `isMobile` 在 `Popup` 与 `Dialog` 间切换
- 统一头部、副标题、说明文案和底部按钮
- 承载 `NodeEditorForm`

输入建议：

- `visible`
- `mode: 'create' | 'edit'`
- `isMobile`
- `form`
- `onClose`
- `onSubmit`
- `onFormChange` 或各字段回调

行为要求：

- 移动端渲染为现有 `timeline-edit-drawer` 视觉结构
- 桌面端不再依赖 antd 默认 `actions` 区，而是使用与移动端一致的自定义内容布局和按钮区，避免再次出现创建/编辑样式漂移

## 页面集成设计

`Timeline.tsx` 调整后保留：

- 创建表单状态 `createForm`
- 编辑表单状态 `editForm`
- 打开/关闭创建节点逻辑
- 打开/关闭编辑节点逻辑
- `handleCreateNode`
- `handleSaveEdit`

`Timeline.tsx` 移除或替换：

- 内联的 `editDialogContent`
- `创建节点` 的内联 `Dialog content`
- 分散的创建/编辑容器 JSX

预期集成方式：

- 渲染一个 `NodeEditorModal` 处理创建
- 渲染一个 `NodeEditorModal` 处理编辑

两者只在以下方面不同：

- `visible`
- `mode`
- `title` / `eyebrow` / `description`
- 表单值来源
- 提交回调

## 样式设计

统一视觉语言基于现有编辑节点实现：

- `timeline-node-dialog`
- `timeline-edit-drawer`
- `timeline-edit-drawer__header`
- `timeline-edit-drawer__body`
- `timeline-edit-drawer__actions`
- `section-label`
- `section-copy`

样式约束：

- 创建与编辑必须共用同一套字段间距、说明区节奏、按钮区布局
- 移动端抽屉和桌面端弹窗允许容器外形不同，但内部内容层级一致
- 不再使用 antd 默认 `Dialog` 的 action table 风格按钮区

## 测试设计

需要更新或补充 `frontend/src/test/timelineModals.test.tsx`：

- 桌面端打开 `创建节点` 时，仍然渲染为 `dialog`
- 移动端打开 `创建节点` 时，渲染为底部抽屉式 `Popup`
- 创建与编辑都使用统一的表单字段与统一按钮布局
- 编辑节点现有测试行为继续成立
- 创建节点提交参数保持不变：
  - `name`
  - `description`
  - `deadline`
- 编辑节点提交参数保持不变：
  - `name`
  - `description`
  - `deadline`
  - `status`

## 风险与缓解

### 风险 1：桌面端 `Dialog` 与自定义内容按钮区叠加

如果仍继续使用 antd `actions`，会导致一部分按钮走默认渲染，一部分按钮走自定义布局，产生双重动作区或样式冲突。

缓解：

- 桌面端 `Dialog` 只作为壳子，不使用默认 `actions`
- 将按钮区并入自定义内容结构内

### 风险 2：移动端和桌面端的 DOM 结构不同导致测试脆弱

现有测试依赖 `role="dialog"`、`Popup` 容器、特定标题文本。如果结构重构但语义保留不完整，测试会变得不稳定。

缓解：

- 保持标题文本稳定
- 保留移动端特征测试点，例如特定测试标识或抽屉容器类名
- 优先测试用户可见行为，不测试实现细节

### 风险 3：创建模式误显示状态字段，改变业务输入面

如果统一表单时直接复用编辑字段全集，可能会把创建流程误改成需要选择状态。

缓解：

- 明确由 `mode` 控制状态字段显隐
- 测试中校验创建模式不出现状态选项

## 验收标准

- 创建节点与编辑节点在视觉结构上保持一致
- 移动端二者都以底部抽屉方式展示
- 桌面端二者都以居中弹窗方式展示
- 创建节点不显示状态字段，编辑节点显示状态字段
- `Timeline.tsx` 中不再保留两套大段重复的节点编辑弹窗 JSX
- 现有创建和编辑业务逻辑、接口参数与成功失败提示不变
