# 婚嫁管理系统完整实现规格

## 1. 项目概述

本规格基于原始设计文档，采用垂直切割方式实现婚嫁管理系统的完整功能。

## 3. 阶段一：时间线 CRUD

### 3.1 后端实现

**新增文件：**
- `routes/timelineRoutes.ts` - 时间线路由
- `controllers/timelineController.ts` - 时间线控制器

**API 接口：**

| 方法 | 路径 | 描述 |
|------|------|------|
| GET | `/api/timeline` | 获取用户的时间线节点列表 |
| POST | `/api/timeline` | 创建新节点 |
| PUT | `/api/timeline/:id` | 更新节点 |
| DELETE | `/api/timeline/:id` | 删除节点 |
| POST | `/api/timeline/update-order` | 更新节点顺序 |

**业务逻辑：**
- 用户首次登录时，自动创建 9 个标准婚嫁节点
- 节点按 `order` 字段排序返回
- 删除节点时，级联删除关联的待办、费用、备忘录、附件

**9 个标准节点初始数据：**
1. 确定结婚意向（order: 1）
2. 双方父母见面（order: 2）
3. 男方上门提亲（order: 3）
4. 彩礼嫁妆三金协商（order: 4）
5. 订婚仪式（order: 5）
6. 婚前筹备（order: 6）
7. 民政局领证（order: 7）
8. 婚礼举办（order: 8）
9. 婚后费用结算收尾（order: 9）

### 3.2 前端实现

**修改文件：**
- `pages/Timeline/Timeline.tsx` - 完善时间线页面

**功能点：**
- 节点列表展示（状态徽章、进度条、截止日期）
- 创建节点表单（名称、描述、截止日期）
- 编辑节点（点击节点进入编辑模式）
- 删除节点（二次确认）
- 节点状态流转（进行中→已完成/已取消）
- 进度百分比计算（基于子待办完成情况）

## 4. 阶段二：节点详情页（5 大功能区）

### 4.1 后端实现

**新增文件：**
- `routes/todoRoutes.ts`、`controllers/todoController.ts`
- `routes/expenseRoutes.ts`、`controllers/expenseController.ts`
- `routes/memoRoutes.ts`、`controllers/memoController.ts`
- `routes/attachmentRoutes.ts`、`controllers/attachmentController.ts`

**API 接口：**

**待办事项（/api/todos）：**
| 方法 | 路径 | 描述 |
|------|------|------|
| GET | `/api/todos?nodeId=:id` | 获取节点的待办列表 |
| POST | `/api/todos` | 创建待办 |
| PUT | `/api/todos/:id` | 更新待办 |
| DELETE | `/api/todos/:id` | 删除待办 |
| PUT | `/api/todos/:id/status` | 更新待办状态 |

**费用记录（/api/expenses）：**
| 方法 | 路径 | 描述 |
|------|------|------|
| GET | `/api/expenses?nodeId=:id` | 获取节点的费用列表 |
| POST | `/api/expenses` | 创建费用记录 |
| PUT | `/api/expenses/:id` | 更新费用记录 |
| DELETE | `/api/expenses/:id` | 删除费用记录 |

**备忘录（/api/memos）：**
| 方法 | 路径 | 描述 |
|------|------|------|
| GET | `/api/memos?nodeId=:id` | 获取节点的备忘录 |
| POST | `/api/memos` | 创建/更新备忘录 |

**附件（/api/attachments）：**
| 方法 | 路径 | 描述 |
|------|------|------|
| GET | `/api/attachments?nodeId=:id` | 获取节点的附件列表 |
| POST | `/api/attachments` | 上传附件（multipart/form-data） |
| DELETE | `/api/attachments/:id` | 删除附件 |
| GET | `/api/attachments/:id/download` | 下载附件 |

**费用分类预设：**
- 收入类：彩礼、礼金、嫁妆回礼、其他收入
- 支出类：婚宴、婚庆、婚车、婚纱摄影、三金/五金、酒店预订、婚车车队、蜜月旅行、其他支出

**待办模板分类（内置）：**
- 彩礼类：沟通彩礼金额、彩礼转账、彩礼确认
- 嫁妆类：嫁妆清单、嫁妆采购
- 三金类：选购三金、购买三金
- 婚宴类：确定婚宴酒店、签订婚宴合同、婚宴菜单确认
- 婚庆类：婚庆公司选择、婚礼策划确认、婚礼现场布置
- 婚车类：婚车预约、婚车路线规划
- 婚纱类：婚纱照拍摄、婚纱礼服租赁/购买

### 4.2 前端实现

**新增文件：**

- 附件列表（图片预览、文件图标、文件名、大小）

- 修改节点名称、描述、截止日期

## 5. 阶段三：协同与统计

### 5.1 WebSocket 实时同步

**Socket.IO 事件设计：**

| 事件名 | 方向 | 数据 | 描述 |
|--------|------|------|------|
| `join_room` | client→server | `{userId, partnerId}` | 加入配对房间 |
| `leave_room` | client→server | `{userId}` | 离开房间 |
| `node_update` | bidirectional | `{type, data}` | 节点更新 |
| `todo_update` | bidirectional | `{type, data}` | 待办更新 |
| `expense_update` | bidirectional | `{type, data}` | 费用更新 |
| `memo_update` | bidirectional | `{type, data}` | 备忘录更新 |
| `attachment_update` | bidirectional | `{type, data}` | 附件更新 |

**同步策略：**
- 用户登录后自动加入配对房间
- 任何数据变更通过 Socket.IO 广播给配对方
- 前端收到推送后自动更新本地状态

## 8. 非功能需求

### 8.1 安全要求
- 所有 API 需要 JWT 认证
- 文件上传校验文件类型和大小（最大 10MB）
- SQL 注入防护（参数化查询）
- XSS 防护（输入转义）

### 8.2 性能要求
- 接口响应时间 < 500ms
- 文件上传速度 < 2s（10MB 以内）
- 前端状态更新无需刷新页面

### 8.3 兼容性
- 移动端优先设计
- 响应式布局支持 320px ~ 1920px
- Ant Design Mobile 组件库保证移动端体验
