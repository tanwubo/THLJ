# 节点工作台重构设计

## 1. 背景与目标

当前节点详情页以多个平级 Tab 组织待办、费用、附件等内容，存在以下问题：

- 费用、附件与待办之间没有真实父子关系，无法表达“某项支出/某个文件属于哪条待办”
- 快捷模板创建出的待办虽然可用，但交互上没有和普通待办完全统一
- 节点与待办的新增/编辑采用内嵌表单，移动端占位突兀，破坏列表阅读流
- 首页头部不固定，长列表滚动时上下文容易丢失
- 日期输入依赖原生输入样式，移动端体验较差

本次重构的目标是：

- 将节点详情页重构为“待办工作台”
- 将费用、附件改为必须从属于某个待办项
- 统一节点、待办的创建和编辑交互为弹窗
- 固定首页头部，仅滚动列表内容
- 优化日期选择体验

## 2. 范围

本次重构包含：

- 后端数据结构调整：费用与附件强关联到待办
- 后端接口调整：增加节点工作台聚合接口，重构费用/附件接口校验
- 前端节点详情页重构为待办工作台
- 前端首页布局调整为固定头部与底部导航
- 前端节点与待办的新增、编辑改为弹窗
- 日期选择组件改造
- 对应自动化测试补充

本次重构不包含：

- 备忘录迁移到待办级，备忘录仍为节点级
- 复杂旧数据自动迁移与归档规则
- 费用或附件的跨待办拖拽迁移
- 大范围视觉改版

## 3. 数据模型设计

### 3.1 待办

`todo_items` 继续作为 `timeline_nodes` 的直接子实体，不改变其主从关系。

### 3.2 费用

`expense_records` 增加必填字段：

- `todo_id INTEGER NOT NULL`

约束要求：

- `todo_id` 必须指向某个真实存在的 `todo_item`
- 该 `todo_item` 必须属于当前访问的 `timeline_node`
- 该 `todo_item` 必须属于当前登录用户可访问的数据范围

### 3.3 附件

`attachments` 增加必填字段：

- `todo_id INTEGER NOT NULL`

约束要求与费用一致。

### 3.4 备忘录

`memos` 保持节点级，不增加 `todo_id`。

### 3.5 删除规则

删除待办项时，后端进行级联删除：

- 删除该待办下所有费用记录
- 删除该待办下所有附件记录
- 如果后续需要处理物理文件删除，也在同一删除链路内执行

这是确定的产品规则，不做“阻止删除”或“迁移后删除”。

## 4. 后端接口设计

### 4.1 新增聚合接口

新增工作台接口：

- `GET /api/timeline/:nodeId/workbench`

返回结构包含：

- 节点基本信息
- 节点下全部待办
- 每个待办关联的费用列表
- 每个待办关联的附件列表
- 节点级备忘录

推荐返回结构：

```json
{
  "node": {
    "id": 1,
    "name": "确定结婚意向",
    "status": "pending",
    "deadline": null
  },
  "todos": [
    {
      "id": 11,
      "nodeId": 1,
      "content": "沟通彩礼金额",
      "status": "pending",
      "deadline": "2026-04-30",
      "expenses": [],
      "attachments": []
    }
  ],
  "memo": {
    "id": 7,
    "nodeId": 1,
    "content": "..."
  }
}
```

这样前端可以一次拿全，避免节点详情页多接口拼装带来的复杂状态同步。

### 4.2 费用接口调整

保留现有费用增删改查接口路径，但其语义改为“待办级费用”：

- 创建费用必须传 `todoId`
- 查询费用时以 `todoId` 为基础，或由工作台接口统一聚合
- 更新、删除费用前必须校验其所属待办及节点归属

如果请求中缺少 `todoId`，返回 `400`。

### 4.3 附件接口调整

附件接口规则与费用一致：

- 上传附件必须传 `todoId`
- 删除附件前必须校验其所属待办及节点归属
- 工作台页优先通过聚合接口读取，不要求前端单独查询每个待办附件

### 4.4 待办删除接口

删除待办的逻辑调整为：

1. 校验待办属于当前节点和当前用户
2. 删除该待办下的附件记录
3. 删除该待办下的费用记录
4. 删除待办本身

这是服务端原子责任，前端不拼装多次删除请求。

### 4.5 数据迁移策略

本次采用开发阶段的结构迁移策略：

- 更新表结构，新增 `todo_id`
- 不实现复杂旧数据自动归档
- 旧的节点级费用/附件测试数据允许失效或由开发者手动清理

原因是当前项目仍处于快速迭代期，复杂迁移逻辑会显著增加实现风险，但不能带来对应价值。

## 5. 前端交互设计

### 5.1 首页

首页改为三段式布局：

- 顶部头部固定
- 中间节点列表区域滚动
- 底部导航固定

用户滚动时，节点列表独立滚动，头部和导航不位移。

### 5.2 节点详情页重构为待办工作台

节点详情页不再以“待办/费用/附件”为平级工作入口，而是以待办为中心组织内容。

页面结构：

- 顶部：返回、节点标题、节点状态
- 主体：待办卡片列表
- 底部或尾部：节点级备忘录、节点操作

每个待办卡片内展示：

- 待办标题
- 截止日期
- 完成状态
- 编辑待办
- 删除待办
- 费用区块
- 附件区块

费用与附件只能从所属待办卡片中新增，不提供节点级全局入口。

### 5.3 快捷模板

快捷模板创建出的待办与普通待办完全等价：

- 可编辑
- 可删除
- 可补充截止日期
- 可挂载费用与附件

快捷模板本质上只是“带预置文案的新建待办入口”，不是特殊类型待办。

### 5.4 节点与待办弹窗

以下场景统一使用弹窗：

- 新增节点
- 编辑节点
- 新增待办
- 编辑待办

不再在列表内部插入表单块。

这样可以保证：

- 移动端主列表结构稳定
- 不因编辑状态打断列表阅读
- 新增与编辑的操作模型一致

### 5.5 删除确认

删除待办时，弹窗明确提示：

- 删除该待办会一并删除其下费用与附件

删除节点时，继续维持危险操作确认。

### 5.6 日期选择器

节点与待办相关日期字段统一改为更友好的弹出式日期选择，不再使用当前原生日期输入的默认视觉。

优先使用现有 `antd-mobile` 体系内的日期选择能力，保持移动端交互一致。

## 6. 状态与数据流

### 6.1 节点工作台加载

进入节点详情页时：

1. 请求 `GET /api/timeline/:nodeId/workbench`
2. 将结果渲染为节点信息 + 待办卡片列表 + 备忘录

### 6.2 待办相关操作

- 新增待办成功后，刷新工作台
- 编辑待办成功后，刷新工作台
- 删除待办成功后，刷新工作台
- 快捷模板创建成功后，刷新工作台

### 6.3 费用与附件相关操作

- 某个待办内新增费用后，刷新工作台或刷新该待办局部数据
- 某个待办内上传附件后，刷新工作台或刷新该待办局部数据
- 删除费用或附件后，同理刷新

当前推荐优先沿用已有“操作后重新拉取”的简单策略，避免在本轮同时引入复杂本地状态合并。

### 6.4 实时同步

已有实时同步基础设施保留。

节点工作台页在待办、费用、附件变更后发出对应事件，并在接收到实时事件时重新拉取工作台数据。这样可以与当前实现风格保持一致。

## 7. 错误处理

### 7.1 前端

- 未选择待办时，不允许创建费用或附件
- 删除待办时，如服务端返回失败，保留当前工作台状态并提示失败原因
- 日期选择为空时按字段可选性校验

### 7.2 后端

- 缺失 `todoId`：返回 `400`
- `todoId` 不属于当前节点：返回 `400` 或 `404`
- `todoId` 不属于当前用户：返回 `403` 或 `404`
- 删除待办时级联删除失败：返回 `500`，不返回部分成功状态

- 创建附件必须带 `todoId`
- 费用/附件的 `todoId` 归属校验
- 删除待办时级联删除费用与附件

补充以下测试：

- 首页固定头部布局行为
- 节点新增、编辑弹窗开关

- 快捷模板创建出的待办可以进入编辑流程

实现完成后必须执行：

## 9. 实施顺序建议

建议按如下顺序实施：

1. 后端 schema 与接口改造
2. 后端工作台聚合接口与级联删除
3. 前端首页固定布局
4. 前端节点详情页重构为工作台
5. 前端节点与待办弹窗
6. 前端费用与附件改为待办上下文
7. 日期选择器优化
8. 全量验证

## 10. 决策摘要

本次重构的核心决策如下：

- 节点详情页改为待办工作台
- 费用与附件必须强关联到待办项
- 删除待办时级联删除其下费用与附件
- 备忘录保持节点级
- 节点、待办的新增与编辑统一为弹窗
- 首页头部固定，仅中间列表滚动
- 日期输入改为更友好的弹出式选择器
