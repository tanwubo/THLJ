# 时间线模板初始化与管理设计

## 背景

当前系统会在两个位置自动为用户注入默认时间线节点：

- 注册成功后，`backend/src/controllers/authController.ts` 会直接插入 9 个默认节点
- 拉取时间线时，`backend/src/controllers/timelineController.ts` 会在用户无节点时自动补齐默认节点

这导致“空时间线”状态在产品中实际上不存在，也就无法为用户提供“从模板初始化”或“从空白开始”的明确选择。

本次改造目标是把“默认时间线”从真实用户数据中抽离出来，升级为独立的“时间线模板”能力。用户在首页首次进入且时间线为空时，可以主动选择一套模板作为初始数据，也可以继续使用现有的“创建节点”能力从空白开始构建自己的时间线。

## 目标

- 取消注册后和首次拉取时间线时的默认节点自动注入
- 引入独立的时间线模板数据模型
- 在用户时间线为空时，在首页提供“选择模板”入口
- 用户可从启用中的模板中选择一套，复制为自己的初始时间线
- 用户也可以完全跳过模板，继续用现有节点创建能力逐个建立时间线
- 提供产品内模板管理页面，仅允许内置管理员账号访问

## 非目标

- 第一版不支持模板节点的截止日期、截止日期偏移、状态预设
- 第一版不支持模板级 todo、费用、备忘录、附件等子数据预置
- 第一版不做复杂 RBAC，不引入完整角色权限系统
- 第一版不做模板复制、模板版本历史、模板分页、模板搜索
- 第一版不单独建设运营后台或顶级后台导航体系

## 核心方案

采用“独立模板表 + 模板节点表 + 专用应用接口”的方案。

模板数据与真实用户时间线节点彻底分离：

- 模板存放在独立表中，支持产品内维护
- 用户真正使用的时间线仍然落在现有 `timeline_nodes` 表中
- 应用模板时，由后端将模板节点复制到当前用户的真实时间线中

这一方案可以满足当前需求，同时为后续扩展模板状态、复制、预览和更复杂的模板内容保留演进空间。

## 数据模型

### 1. `timeline_templates`

新增时间线模板表，用于保存模板基本信息。

建议字段：

- `id`
- `name`
- `description`
- `is_active`
- `created_at`
- `updated_at`

职责：

- 描述一个可供用户选择的模板
- 支持启用 / 停用
- 作为模板节点的父实体

### 2. `timeline_template_nodes`

新增模板节点表，用于保存某一模板下的节点定义。

建议字段：

- `id`
- `template_id`
- `name`
- `description`
- `order`
- `created_at`
- `updated_at`

职责：

- 定义模板内的节点顺序
- 保存节点名称和预置描述

### 3. 第一版模板节点字段范围

模板节点只包含：

- 节点名称
- 预置描述
- 排序

模板节点不包含：

- 截止日期
- 截止日期偏移
- 默认状态
- todo 模板
- 费用模板
- memo 模板
- attachment 模板

## 权限模型

### 1. 管理端权限

模板管理能力只允许“内置管理员账号”访问。

第一版不新增角色表或权限表，改为后端通过内置管理员用户名白名单判断当前用户是否为管理员。该白名单可通过常量或环境变量维护。

### 2. 前端权限展示

后端在登录响应和 `/api/auth/profile` 中返回 `isAdmin`。

前端据此控制：

- 是否展示“时间线模板管理”入口
- 是否允许进入模板管理页面

但前端展示只作为体验控制，真正权限校验以后端接口为准。

### 3. 普通用户权限

普通用户仅允许：

- 读取启用中的模板列表
- 查看模板详情和节点预览
- 在自己的时间线为空时应用模板

普通用户不允许：

- 创建模板
- 编辑模板
- 删除模板
- 上下架模板

## 后端接口设计

### 1. `GET /api/timeline-templates`

用途：

- 首页模板选择列表
- 管理员模板管理列表

行为：

- 普通用户默认只返回启用中的模板
- 管理员可通过 `?includeInactive=1` 获取全部模板

返回内容：

- 模板基础信息列表
- 可附带节点数量等轻量统计信息

### 2. `GET /api/timeline-templates/:id`

用途：

- 首页模板预览
- 管理页模板编辑

返回内容：

- 模板基础信息
- 模板节点列表，按 `order` 排序

### 3. `POST /api/timeline-templates`

仅管理员可调用。

用途：

- 创建模板

请求内容：

- 模板名称
- 模板描述
- 是否启用
- 模板节点数组

### 4. `PUT /api/timeline-templates/:id`

仅管理员可调用。

用途：

- 更新模板基本信息
- 覆盖保存整套模板节点内容

请求内容：

- 模板名称
- 模板描述
- 是否启用
- 模板节点数组

节点保存策略：

- 第一版采用整套覆盖更新，避免复杂增量 diff 逻辑

### 5. `DELETE /api/timeline-templates/:id`

仅管理员可调用。

用途：

- 删除模板及其模板节点

### 6. `POST /api/timeline/apply-template`

普通用户和管理员都可调用，但只允许对“当前真实时间线为空”的数据空间生效。

请求内容：

- `templateId`

应用规则：

- 模板必须存在
- 模板必须为启用状态
- 当前 `dataOwnerId` 对应的真实时间线必须为空
- 模板节点按顺序复制到 `timeline_nodes`
- 新生成的节点字段规则：
  - `name` 来自模板节点
  - `description` 来自模板节点
  - `order` 按模板节点顺序
  - `status = pending`
  - `deadline = null`

拒绝条件：

- 模板不存在
- 模板未启用
- 当前时间线非空

## 现有逻辑改造

### 1. 移除注册默认节点注入

修改 `backend/src/controllers/authController.ts`：

- 删除注册成功后为新用户插入 9 个默认节点的逻辑

注册完成后：

- 仅创建用户
- 设置 `data_owner_id`
- 返回 token 与用户信息

### 2. 移除拉取时间线时的自动补齐

修改 `backend/src/controllers/timelineController.ts`：

- 删除 `getTimeline` 中“无节点时自动插入默认节点”的逻辑

新的行为：

- 用户无节点时返回空数组
- 前端根据空数组决定是否展示模板入口与空状态

### 3. 共享数据场景

系统当前已通过 `dataOwnerId` 实现情侣绑定后的共享数据读取。

模板应用时必须沿用该约定：

- 模板复制目标为当前请求上下文中的 `dataOwnerId`
- 只要共享时间线中已有任意节点，即视为“非空”，禁止再次应用模板

这样可以避免情侣绑定后重复套模板，导致共享数据被重复初始化。

## 首页交互设计

### 1. 触发条件

当 `GET /api/timeline` 返回的 `nodes.length === 0` 时，首页进入“空时间线状态”。

### 2. 首页空状态内容

时间线首页空状态提供两个动作：

- `选择模板`
- `创建第一个节点`

行为约束：

- “创建第一个节点”继续复用现有节点创建流程，不改现有能力
- “选择模板”打开模板选择弹层或抽屉

### 3. 模板选择流程

用户点击“选择模板”后，前端展示：

- 模板列表卡片
- 当前模板简介
- 模板节点预览列表
- `使用此模板` 按钮

应用成功后：

- 关闭模板选择层
- 重新请求时间线
- 首页切换到正常节点列表态

### 4. 入口隐藏规则

只要用户已经拥有任意真实节点，无论节点来自模板还是手动创建：

- 首页不再展示“选择模板”入口

这满足“模板仅作为初始化能力”的产品边界，避免用户在已有真实数据后重复导入模板。

## 模板管理页面设计

### 1. 入口位置

不新增后台导航体系。

模板管理入口放在现有设置页中：

- 仅当 `isAdmin === true` 时显示“时间线模板管理”入口

点击后进入单独页面，例如：

- `/settings/timeline-templates`

### 2. 页面能力

模板管理页面包含以下能力：

- 查看模板列表
- 创建模板
- 编辑模板名称和描述
- 启用 / 停用模板
- 删除模板
- 管理模板节点的增删改和排序

### 3. 节点排序交互

第一版不引入复杂拖拽。

可接受的简化实现：

- 使用“上移 / 下移”调整顺序
- 或按当前编辑列表顺序整体保存

原则是先保证模板管理可用，减少实现和测试复杂度。

## 初始化与种子策略

### 1. 系统默认模板

系统启动后检查模板表是否为空：

- 如果为空，则写入一套内置默认婚礼模板
- 该模板包含当前现有的 9 个标准婚礼节点

这套默认模板进入模板表，而不是进入真实用户时间线。

### 2. 避免重复注入

种子逻辑只在“模板表为空”时执行一次，避免每次启动都重复写入。

### 3. 后续维护

模板初始化完成后：

- 管理员可在产品内继续编辑这套默认模板
- 也可新建其他模板

## 前端改造范围

### 1. 时间线页

修改 `frontend/src/pages/Timeline/Timeline.tsx`：

- 保留现有创建节点流程
- 空状态增加“选择模板”入口
- 模板应用成功后刷新时间线
- 非空时不展示模板入口

### 2. API 层

修改 `frontend/src/services/api.ts`：

- 增加模板列表、模板详情、模板管理、应用模板等 API 定义
- 为 `User` 增加 `isAdmin`

### 3. 设置页

修改 `frontend/src/pages/Settings/Settings.tsx`：

- 当 `isAdmin` 为真时展示模板管理入口

### 4. 新页面 / 组件

建议新增：

- 模板选择弹层或抽屉组件
- 模板管理页
- 模板编辑表单或节点列表编辑组件

具体文件命名可遵循现有前端页面结构，但应保持职责清晰，避免将模板选择和模板管理混杂在同一大文件中。

## 后端改造范围

建议新增：

- 模板控制器
- 模板路由
- 模板服务或模板相关数据库操作封装

并修改：

- `authController`
- `timelineController`
- `auth/profile` 返回结构
- 数据库初始化 / 引导逻辑

## 错误处理要求

### 1. 应用模板失败

需给出明确错误语义：

- 模板不存在
- 模板已停用
- 当前时间线已有数据，无法重复应用

前端应将这些错误直接透出为用户可理解的提示。

### 2. 管理端权限失败

管理员接口若被普通用户调用，应返回明确的权限错误。

### 3. 并发安全

模板应用时应在后端以单次受控流程完成：

- 先检查真实时间线是否为空
- 再写入模板节点

避免用户多次点击导致重复初始化。

## 测试要求

### 1. 后端测试

需要覆盖：

- 注册后不再自动创建默认节点
- 获取空时间线时返回空数组
- 普通用户只能读取启用模板
- 管理员可查看全部模板
- 管理员可创建、更新、删除、启停模板
- 空时间线用户可成功应用模板
- 非空时间线用户不能再次应用模板
- 绑定情侣后，应用模板基于 `dataOwnerId` 判断空状态

### 2. 前端测试

需要覆盖：

- 时间线为空时出现“选择模板”和“创建第一个节点”
- 时间线非空时不显示“选择模板”
- 模板选择弹层可展示模板预览节点
- 应用模板成功后切换为真实时间线列表
- 非管理员设置页不显示模板管理入口
- 管理员可进入模板管理页并完成模板保存

## 实施顺序

建议按以下顺序推进：

1. 数据库与模板种子初始化
2. 后端模板接口与管理员权限校验
3. 移除注册与拉取时间线中的默认节点注入
4. 首页空状态模板选择能力
5. 设置页管理员入口与模板管理页
6. 测试补齐与回归验证

这一顺序可以将“默认注入下线”和“模板初始化上线”串成一个连续交付链路，降低中途出现不可用空档的风险。

## 成功标准

- 新注册用户首次进入首页时，时间线为空，不自动生成默认节点
- 空时间线首页可选择模板或继续手动创建节点
- 用户应用模板后获得一套真实时间线节点
- 真实时间线一旦非空，模板入口不再展示
- 管理员可在产品内维护模板
- 普通用户无法访问模板管理能力
